# AGENTS.md

## 文档约定

项目文档正文使用**中文**编写，便于团队理解与协作。

- 文件名可使用英文；API 路径、组件名、类型名、配置项等技术标识保持英文原文。
- 新增或更新文档时，优先写清业务背景、业务流程、数据关系和验证方式，避免只罗列代码路径。
- 修改实现前，先阅读根目录中与 Shopify 定制业务相关的需求、数据模型和方案文档；需求冲突时，以最新需求说明和用户本次指令为准，并在交付说明中记录取舍。

## 项目边界

本文件所在目录即当前 Cloudflare/Node.js 项目根目录。约定使用 `./` 表示该目录，不得将项目根目录写死为某个仓库名、软链接名或上级路径。

```text
./                   # Node.js 22 + TypeScript + React/Next.js API + vinext/Vite
                     # 部署到 Cloudflare Worker，使用 D1/Drizzle
Shopify Theme 项目   # 若任务明确涉及主题，以用户指定或实际发现的主题目录为准
```

- 当前项目根目录 `./` 是管理端、Storefront API 和持久化实现的核心 Node 项目。
- Shopify 主题侧目录不得在本文中写死；任务涉及主题时，先根据用户指令和工作区实际结构定位。主题项目遵循其现有目录约定，不引入 React、Node 服务端依赖或打包器运行时。
- 不跨边界复制业务逻辑。管理配置、校验和持久化放在当前项目根目录；主题侧仅负责展示、交互以及调用公开 Storefront API。
- 未经明确要求，不修改生成目录或本地状态目录，如 `node_modules/`、`dist/`、`.next/`、`.wrangler/`。

## AI 强制执行约定

### 开始工作前

1. 阅读本文件、当前项目根目录的 `README.md` 及与任务相关的业务文档。
2. 检查 `git status --short`。工作区可能包含用户未提交内容，必须保留无关改动，不得擅自清理、覆盖或回退。
3. 阅读目标文件及其直接调用方、数据类型、API 路由和数据库定义，禁止只根据文件名猜测实现。
4. 以当前项目根目录 `package.json` 的 `engines`、依赖版本和 scripts 为准；不要擅自切换包管理器或升级依赖。

### 变更原则

- 仅修改完成任务所必需的文件，避免顺手重构、批量格式化或重写无关代码。
- 优先沿用现有依赖和平台能力；新增生产依赖前，应确认现有能力无法满足需求，并说明原因。
- TypeScript 代码保持 ESM，使用明确类型；不得用 `any`、非空断言或跳过类型检查掩盖模型问题。
- 服务端代码不得依赖 Node.js 专属 API，除非 Cloudflare Worker 的 `nodejs_compat` 已明确支持且确有必要。
- 面向外部的行为变更必须同步更新相关 README、业务文档或 API 说明。

## Node.js、React 与 TypeScript 规范

- Node.js 版本遵循当前项目根目录 `package.json` 的 `engines` 配置。
- 在当前项目根目录使用 `npm` 和已存在的 scripts；仓库未声明其他包管理器时，不新增其他锁文件。
- React 组件默认采用函数组件。仅在需要交互、浏览器 API 或客户端状态时使用客户端组件；数据访问和密钥处理留在服务端。
- 共享业务类型、校验和数据访问逻辑应提取到明确模块，避免在多个 route 或组件中复制。
- 外部输入在系统边界处完成解析和校验，包括 API body、path/query 参数、Shopify 数据及数据库 JSON 字段。
- JSON 解析失败、资源不存在、业务校验失败和服务端异常应返回可区分的状态码与稳定错误结构，不向客户端泄露堆栈、SQL 或密钥。
- 日期时间统一使用 ISO 8601 UTC 字符串；金额不得使用浮点数进行精确计算，优先使用最小货币单位整数。

## API 与分层规范

标准调用方向保持清晰：

```text
React / Shopify Theme
        ↓
app/api/**/route.ts
        ↓
共享业务服务与校验
        ↓
D1 / 外部 Shopify API
```

- `route.ts` 负责 HTTP 边界：读取请求、鉴权、校验、调用业务逻辑并构造响应；复杂业务规则和 SQL 不应堆积在路由文件中。
- API 路径使用全小写 kebab-case；资源路径优先使用名词，动作型端点仅用于 `publish`、`validate` 等无法自然表达为资源状态的操作。
- 所有写接口必须考虑鉴权、权限、幂等性、并发更新和审计信息；当前 POC 若暂未实现，必须在交付说明或文档中明确风险，不得将其描述为生产可用。
- Storefront API 必须最小化返回字段，不暴露草稿配置、内部标识或管理信息。
- CORS 不得在生产环境使用无条件 `*`。允许来源必须由可信配置限定，并正确处理预检请求。
- 修改 API 时同步检查管理端调用、主题侧调用、README 中的接口清单和相关业务文档。

## D1 与 Drizzle 持久层规范

涉及表结构、查询、迁移或 JSON 字段时，必须同时检查：

- `db/schema.ts`；
- `src/repositories/database.ts` 及相关 Repository；
- 所有使用相关表的 API route。

强制要求：

- Drizzle schema、D1 实际建表 SQL 和 TypeScript 行类型必须保持一致，字段命名转换要显式且可追踪。
- 所有 SQL 使用绑定参数，禁止拼接用户输入。
- 表结构变更必须可重复执行，并兼容已有数据；破坏性迁移须先说明备份、回滚和数据转换方案。
- JSON 字段写入前校验结构，读取时处理无效或旧版本数据；不得把未经校验的任意对象直接作为已发布配置。
- 多步且要求原子性的写操作使用 D1 支持的事务或 batch 语义，并明确失败行为。
- 新增查询应评估索引、唯一约束和分页，避免无界全表读取。
- 不在正常请求中长期承担复杂迁移。POC 阶段的 `ensureDb` 若继续使用，新增迁移仍须保证幂等，并记录未来迁移到正式 migration 的条件。

## Shopify Theme 规范

- 遵循 Shopify 标准目录职责：Liquid 负责服务端模板，`assets/` 放静态 JS/CSS，`sections/`、`blocks/` 和 `snippets/` 保持可配置与可复用。
- Liquid 输出进入 HTML、属性、URL 或 JavaScript 上下文时，使用适合该上下文的转义或 JSON 序列化，防止 XSS 和语法破坏。
- 主题 JavaScript 使用原生 ESM/Custom Elements 等现有模式，不污染全局作用域，不覆盖 Shopify 标准事件和表单行为。
- 定制数据写入购物车时使用稳定、可追踪的 line item properties；显示字段与内部字段命名要区分，并与数据模型文档一致。
- 事件监听必须可重复初始化并可清理，兼容 Shopify Theme Editor 的 section 动态加载/卸载。
- 修改主题交互后，至少验证产品页初始化、变体切换、定制校验、加入购物车、购物车回显和移动端布局。

## 配置与安全

- 密钥、Token、店铺域名和数据库标识通过环境变量或平台绑定提供，禁止写入源码、文档示例、日志或提交到 Git。
- 不读取、展示或修改与任务无关的 `.env*` 内容；新增变量时只更新不含真实值的示例和说明。
- 日志不得包含 Cookie、Authorization、用户隐私数据、完整定制尺寸或外部 API 密钥。
- 管理端写接口上线前必须具备可靠鉴权与权限校验；不能仅依赖页面入口隐藏。
- 对 Shopify Webhook 或外部回调，必须验证签名、限制请求体大小，并考虑重放与幂等处理。

## 验证与交付

修改当前 Cloudflare/Node.js 项目后，根据变更范围在项目根目录执行：

```bash
npm run lint
npm test
```

- `npm test` 当前等价于生产构建，是最低限度的类型、打包和 Worker 兼容性检查。
- 数据库或 API 变更还必须验证成功路径、无效输入、资源不存在和存储失败；有自动化测试框架时补充回归测试。
- 主题变更应使用 Shopify 主题检查工具（若项目已配置）并完成相关浏览器人工验证。
- 若受环境、凭据或外部服务限制无法执行某项验证，必须明确列出“未执行的验证、原因和建议命令”，不得声称已通过。
- 最终交付说明应简要列出变更文件、行为变化、已执行验证及仍存在的风险。
